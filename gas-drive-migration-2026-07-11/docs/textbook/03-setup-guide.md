## 3.1 【移行先の管理者】組織外ユーザーの参加を許可する

移行先 (`@misugi-corp.co.jp`) の Google Workspace **特権管理者**が設定する。

1. [管理コンソール](https://admin.google.com) にログイン
2. `アプリ` → `Google Workspace` → `ドライブとドキュメント` → `共有設定` を開く
3. 対象の組織部門で以下を確認・変更する
   - **共有オプション**: 組織外との共有を「オン」にする
   - **共有ドライブの作成**: 「組織外のユーザーに共有ドライブ内のファイルへのアクセスを許可する」に相当する項目を有効にする
     (この設定が無効だと、次節でメンバー追加しようとしても移行元アカウントを追加できない)

> 💡 設定の反映には最大 24 時間かかることがある (通常は数分)。

<details>
<summary>📖 用語解説: 管理コンソール / 特権管理者</summary>

管理コンソール (admin.google.com) は Google Workspace 組織全体の設定を行う
管理者専用画面。特権管理者はその組織の最上位権限を持つアカウントで、
共有ポリシーの変更はこの権限が必要。一般ユーザーの画面には出てこない。

</details>

## 3.2 【移行先ユーザー】共有ドライブの準備とメンバー追加

移行先ドメインのユーザー (または管理者) が行う。

1. [Google ドライブ](https://drive.google.com) → 左メニュー `共有ドライブ` → 新規作成 (既存を使うならそれで良い)
2. 共有ドライブを開き、`メンバーを管理` を開く
3. 移行元アカウント (例: `xxxx@google.com`) を追加し、役割を **「コンテンツ管理者」** にする
   - 「投稿者」ではフォルダ作成・移動が制限される場合があるため不可
   - 追加できない場合は 3.1 の管理者設定が未完了
4. (必要なら) 共有ドライブ内に移行先の親フォルダを作っておく
   - 共有ドライブ直下に置くなら不要

## 3.3 【移行元ユーザー】GAS プロジェクトを作成する

以降はすべて**移行元アカウント (`@google.com` 側)** のブラウザで行う。

1. <https://script.new> を開く (自動的に新しい Apps Script プロジェクトが作られる)
   - または <https://script.google.com> → `新しいプロジェクト`
2. プロジェクト名を分かりやすいものに変更する (例: `drive-migration`)

## 3.4 Drive API (高度なサービス) を有効化する

このスクリプトは Drive API v3 を「高度なサービス」として使う。

**方法A: エディタから有効化 (かんたん)**

1. エディタ左側の `サービス` の `+` をクリック
2. 一覧から `Drive API` を選び、**バージョンに `v3`** を指定して `追加`
   - 識別子 (userSymbol) は `Drive` のまま

**方法B: マニフェストごと差し替え (確実・推奨)**

1. エディタ左の `プロジェクトの設定` (歯車) → `「appsscript.json」マニフェスト ファイルをエディタで表示する` にチェック
2. エディタに `appsscript.json` が現れるので、このリポジトリの
   [`dist/appsscript.json`](../../dist/appsscript.json) の内容をそのまま貼り付けて保存

方法Bなら Drive API v3 の有効化・タイムゾーン・必要スコープが一度に設定される。

<details>
<summary>📖 用語解説: マニフェスト (appsscript.json)</summary>

GAS プロジェクトの設定ファイル。使用する高度なサービス、要求する OAuth スコープ、
タイムゾーンなどを JSON で宣言する。通常は隠れているが、設定で表示できる。

</details>

<details>
<summary>📖 用語解説: スコープ (OAuth スコープ)</summary>

スクリプトがユーザーに要求する権限の範囲。本ツールは
`auth/drive` (ドライブの読み書き)・`auth/script.scriptapp` (トリガー作成)・
`auth/script.send_mail` (完了メール通知) の3つを使う。初回実行時の承認画面に
この内訳が表示される。

</details>

## 3.5 コードを配置する

**方法A: コピー&ペースト (最短)**

1. このリポジトリの [`dist/main.js`](../../dist/main.js) を開き、全文コピー
2. GAS エディタの `コード.gs` の中身をすべて消して貼り付け、保存 (`Ctrl+S`)

**方法B: clasp でローカルから push (開発者向け)**

コードを修正しながら使う場合はこちらが便利。手順は [第5章](./05-dev-environment.md#54-clasp--ローカルから-gas-へデプロイ) を参照。

## 3.6 CONFIG (設定変数) を書き換える

`main.js` 冒頭の `CONFIG` を自分の環境に合わせて書き換える。各変数の意味:

| 変数 | 必須 | 説明 |
| --- | --- | --- |
| `SOURCE_FOLDER_ID` | ★ | **移行元フォルダの ID**。マイドライブで対象フォルダを開いた URL `https://drive.google.com/drive/folders/`**`1AbCdEfGh...`** の太字部分 |
| `DEST_FOLDER_ID` | ★ | **移行先の ID**。共有ドライブそのものの ID (共有ドライブ直下へ置く場合) でも、共有ドライブ内フォルダの ID でもよい。取り方は同じく URL の `folders/` 以降。マイドライブ内のフォルダを指定すると安全装置がエラーで止める |
| `CREATE_TOP_FOLDER` | | `true` (既定): 移行先に移行元と同名のトップフォルダを作ってその中へ移行 / `false`: 移行先フォルダ直下に中身を直接展開 |
| `DRY_RUN` | | `true` (既定): **一切変更せず**、実行予定の操作をログに出すだけのお試しモード。本実行時に `false` へ |
| `COPY_FALLBACK` | | 移動に失敗したファイル (他人がオーナー等) をコピーで救済するか。既定 `true`。コピーはファイル ID が変わる点に注意 |
| `TRASH_ORIGINAL_AFTER_COPY` | | コピー救済後、自分がオーナーの元ファイルをゴミ箱に入れるか。既定 `false` (残す) |
| `NOTIFY_EMAIL` | | 完了・エラー停止時の通知先メールアドレス。空文字 `''` なら通知しない |
| `TIME_LIMIT_MS` | | 1回の実行の制限時間。既定 4.5分 (GAS の上限約6分より手前で自動中断するため) 。通常変更不要 |
| `RESUME_DELAY_MS` | | 自動再開までの待ち時間。既定 60秒。通常変更不要 |
| `PAGE_SIZE` | | 一覧取得の1ページ件数 (最大1000)。通常変更不要 |
| `MAX_RETRIES` | | 一時エラー時のリトライ回数。既定 5。通常変更不要 |

> 💡 **ID の取り方**: ブラウザで対象フォルダ (または共有ドライブのトップ) を開き、
> アドレスバーの `folders/` の後ろの英数字列をコピーするだけ。
> 共有ドライブ自体の ID は、共有ドライブのトップ画面を開いたときの URL に出る。

## 3.7 DRY_RUN で計画を確認する

1. エディタ上部の関数選択プルダウンで **`startMigration`** を選ぶ
2. `実行` をクリック
3. **初回のみ権限の承認画面**が出る:
   - `権限を確認` → 自分のアカウントを選択
   - 「このアプリは Google で確認されていません」と警告が出るが、これは
     **自作スクリプトなので当然の表示**。`詳細` → `(プロジェクト名)(安全ではないページ) に移動` → 内容を確認して `許可`
4. 下部の `実行ログ` に、作成予定フォルダと移動予定ファイルが `[DRY_RUN]` 付きで流れる
5. 件数や階層がイメージ通りか確認する

<details>
<summary>📖 用語解説: 「このアプリは Google で確認されていません」警告</summary>

Google の審査を受けていない OAuth アプリに出る標準の警告。ストア公開アプリでは
ないから出るだけで、コードは自分の手元にある本リポジトリのものなので、内容を
確認した上で許可すればよい。逆に言えば、**出所不明のスクリプトにこの承認を
与えてはいけない** (ドライブ全体を操作できる権限のため)。

</details>

## 3.8 本実行する

1. `CONFIG.DRY_RUN` を `false` に変更して保存
2. もう一度 `startMigration` を実行
3. あとは自動で進む:
   - 約4.5分ごとに自動中断し、約60秒後に自動再開する (何もしなくてよい)
   - 進捗はいつでも **`printStatus`** を実行するとログで確認できる
   - 再開の予約状況はエディタ左の `トリガー` (時計アイコン) で見られる
4. 完了すると `NOTIFY_EMAIL` にレポートが届く (設定した場合)

> ⚠ 実行中は移行元フォルダ内のファイルの追加・移動を控えること。
> 移行中に追加されたファイルは拾われる場合と漏れる場合がある
> (完了後にもう一度 `startMigration` すれば差分だけ移行される → 第4章「べき等性」)。

途中で止めたくなったら:

| やりたいこと | 実行する関数 |
| --- | --- |
| 一時停止 → 後で続き | `cancelMigration` ではなく**放置でよい** (自動中断される)。手動再開は `resumeMigration` |
| 完全に中止 | `cancelMigration` (移動済みファイルはそのまま残る) |
| 状態を消してやり直し | `resetState` → `startMigration` |

## 3.9 完了確認と検証

1. `printStatus` でレポートを確認: `状態: DONE`、`失敗: 0` が理想
2. 失敗がある場合は一覧の理由を確認 (対処法は [第6章](./06-operations.md))
3. 移行元フォルダに (自分がオーナーの) ファイルが残っていないことを確認
4. 移行先の共有ドライブで階層とファイルを開いて確認

## 3.10 後片付け (任意)

移行後、移行元には**空のフォルダの殻**が残る (フォルダは移動ではなく作り直しのため)。
きれいにしたければ:

1. `DRY_RUN: true` のまま `trashEmptySourceFolders` を実行し、削除予定を確認
2. `DRY_RUN: false` にして再実行

このユーティリティは**完全に空のフォルダしかゴミ箱に入れない**ため、
移行漏れファイルを巻き込む心配はない。
