# 第 5 章: 実行手順(運用マニュアル)

この章は、実際に所有権を譲渡するときの手順書です。**上から順に、飛ばさずに**実施してください。特に DRY RUN(Step 2)を飛ばして本番実行してはいけません。

## 5.0 実行前チェックリスト

| # | 確認事項 |
| --- | --- |
| 1 | 譲渡先ユーザーは自分と**同じドメイン**の Google Workspace アカウントか |
| 2 | 譲渡先ユーザーに引き継ぎの合意を取ったか(相手のストレージ容量も消費されます) |
| 3 | 組織の管理者ポリシーで所有権譲渡が禁止されていないか(不明なら小さなフォルダで試す) |
| 4 | `config.ts` の `dryRun` が `true` のままか(最初は必ず true) |
| 5 | 対象範囲は正しいか(`rootFolderId` を指定したか、マイドライブ全体でよいか) |

## 5.1 Step 0: 規模を把握する

エディタ(`mise run open` で開く)の関数選択メニューから **`countOwnedFiles`** を選んで実行します。

```
自分が所有するアイテム: ファイル 823 件 / フォルダ 141 件
```

のようなログが出ます(1,000 件で打ち切り)。所要時間の見積もり(概ね数千件 / バッチ)と、実行後の突き合わせに使うので、値を控えておいてください。

<details>
<summary>📘 用語解説: 実行ログの見方</summary>

Apps Script エディタで関数を実行すると、画面下部に「実行ログ」ペインが開き、`console.log` の出力がリアルタイムに表示されます。過去の実行分は、左メニューの「実行数」(時計アイコン)から一覧でき、各実行を開くとログを確認できます。トリガーによる自動実行(自分がエディタを閉じていた間の分)のログも「実行数」から見られます。より高度な検索は Cloud Logging でも可能です。

</details>

## 5.2 Step 1: 設定を書き換えて push する

`src/config.ts` を編集します。

```typescript
newOwnerEmail: 'successor@your-domain.com', // 譲渡先(同一ドメイン)
rootFolderId: '1AbC...xyz',                 // 対象フォルダの ID('' なら全体)
dryRun: true,                               // まだ true のまま!
```

フォルダ ID は、Drive でフォルダを開いた URL の `https://drive.google.com/drive/folders/【この部分】` です。

編集したら反映します。

```bash
$ mise run push
```

## 5.3 Step 2: DRY RUN で対象を確認する

エディタで **`startTransfer`** を実行します(初回は承認画面 → 第 2 章 2.9)。ログに次のような出力が流れます。

```
所有権の一括譲渡を開始します: you@your-domain.com → successor@your-domain.com(ツリー走査)
[DRY RUN] 実際の譲渡は行いません。対象の列挙とログ出力のみ行います。
[DRY RUN] 譲渡対象: ファイル「議事録_2025-04.docx」(id: 1a2B...)
[DRY RUN] 譲渡対象: フォルダ「プロジェクトA」(id: 9z8Y...)
...
=== すべての処理が完了しました ===
モード: ツリー走査 (DRY RUN)
譲渡先: successor@your-domain.com
バッチ回数: 1
走査済み: 964 件
譲渡対象 (DRY RUN): 810 件
スキップ(自分の所有物でない): 154 件
エラー: 0 件
```

**確認すべきポイント:**

- 譲渡先メールアドレスは正しいか
- 「譲渡対象」の件数は Step 0 の感覚と合っているか(想定外に多い/少ないなら `rootFolderId` を疑う)
- 譲渡対象の中に「引き継ぎたくない私物」が混ざっていないか(混ざっていたら対象フォルダを分ける)

> 💡 対象が多い場合、DRY RUN でも 4.5 分ごとに中断 → 自動再開します。「制限時間が近づいたため、いったん中断しました」のログが出たら、放置しておけば約 60 秒後に続きが自動実行されます。全体の完了は後述の `showStatus` や「実行数」画面で確認します。

## 5.4 Step 3: 本番実行

DRY RUN の結果に問題がなければ、`src/config.ts` の 1 箇所だけを変えます。

```typescript
dryRun: false, // 本番モード
```

```bash
$ mise run push
```

そしてエディタで、もう一度 **`startTransfer`** を実行します。今度は実際に譲渡が行われ、ログが `[DRY RUN] 譲渡対象:` から `譲渡完了:` に変わります。

> ⚠️ ここから先、処理されたファイルの所有権は実際に移っています。途中で `stopTransfer()` しても、**それまでに譲渡した分は戻りません**。

## 5.5 Step 4: 長時間ジョブの見守り方

対象が数千件を超えると、完了まで「4.5 分実行 + 60 秒待機」のサイクルを何度も繰り返します(1 万件で 1〜2 時間程度が目安。ファイルサイズには依存しません)。実行状態がどう遷移するかは第 3 章の図をそのまま使えます。

![実行状態の遷移図](./images/04_state_machine.svg)

*図 5-1(再掲): 今どの状態にいるかは showStatus() とトリガー画面で確認できる*

見守りの道具は 3 つあります。

1. **`showStatus()` を実行**: 保存されている進捗(バッチ回数・処理済み件数・残キュー数)をログに表示します。「実行中の処理はありません」と出たら、完了して状態がクリアされた後です
2. **エディタ左メニューの「実行数」**: バッチごとの実行履歴(所要時間・成否・ログ)が並びます。`resumeTransfer` が定期的に成功し続けていれば健全です
3. **エディタ左メニューの「トリガー」**: 次の再開予約(`resumeTransfer` の時間主導トリガー)が 1 件あるはずです。処理完了後は自動で消えます

**エディタやパソコンは閉じてかまいません。** 再開はすべて Google のサーバー側でトリガーが行います。

## 5.6 Step 5: 完了確認

1. `showStatus()` → 「実行中の処理はありません。」
2. 「実行数」の最後の実行ログに「=== すべての処理が完了しました ===」とサマリが出ている(`エラー: 0 件` かも確認)
3. `countOwnedFiles()` を再実行 → 対象範囲のファイルを譲渡し終えていれば、件数が大きく減っている
4. Drive の画面でいくつかのファイルを開き、オーナーが譲渡先になっていることを目視確認
5. 仕上げに `startTransferAllOwned()`(検索走査)を DRY RUN で実行すると、ツリーの外にあった取り残し(他人のフォルダ内の自分ファイル等)を洗い出せます

エラーが残った場合は、ログから `譲渡失敗:` の行を拾い、[5.8 トラブルシューティング](#58-トラブルシューティング)で原因を特定して対処します。エラーになったファイルは所有権が移っていないので、原因解消後にもう一度実行すれば拾い直せます(処理済みのものは「スキップ」されるだけで無害です)。

## 5.7 知っておくべきクォータ(上限)

| 項目 | 上限 | 本ツールとの関係 |
| --- | --- | --- |
| 1 回の実行時間 | 6 分 | 4.5 分で自主的に中断して回避 |
| トリガー合計実行時間 | 90 分/日(個人)、6 時間/日(Workspace) | 超巨大な Drive では 1 日で終わらないことがある。翌日 `resumeTransfer()` を手動実行すれば続きから再開できる |
| トリガー数 | 20 個/ユーザー/スクリプト | 使い捨てトリガーを毎回掃除して回避(3.7 節) |
| スクリプトプロパティ 1 値 | 9KB | チャンク分割で回避(4.6 節) |
| スクリプトプロパティ合計 | 500KB | キューが極端に長い(数万フォルダ)場合の理論上の壁。実用上はまず到達しない |

最新の正確な値は公式の [Quotas for Google Services](https://developers.google.com/apps-script/guides/services/quotas) を参照してください。

## 5.8 トラブルシューティング

| 症状 | 原因 | 対処 |
| --- | --- | --- |
| `push` 時に `User has not enabled the Apps Script API` | Apps Script API が無効 | [設定ページ](https://script.google.com/home/usersettings)でオンにして数分待つ(2.4 節) |
| `push` 時に `Manifest file has been updated. Do you want to push and overwrite?` | マニフェストの上書き確認 | `y` を入力(手元が正)。毎回聞かれるのが嫌なら `npx clasp push -f` |
| `.clasp.json` が見つからないと言われる | プロジェクト未紐付け | 2.6 節の方法 A または B を実施 |
| 実行時に `未完了の処理が残っています` | 前回の実行が中断状態のまま | 続きをやるなら `resumeTransfer()`、やり直すなら `stopTransfer()` してから開始 |
| 実行時に `ロックを取得できませんでした` | 別の実行(トリガー等)が進行中 | 数分待って再実行。「実行数」で進行中の実行を確認 |
| `譲渡失敗: ...` が特定ファイルで出る | ドメイン外への譲渡、管理者ポリシー、特殊なファイル種別など | エラーメッセージを確認。ドメイン・ポリシー要因なら管理者に相談。当該ファイルだけ手動対応でも可 |
| 譲渡したはずのファイルがまだ自分のオーナーで見える | Drive 画面のキャッシュ | 再読み込み。反映に少し時間がかかることもある |
| `状態データが壊れています` | プロパティの手動編集や保存途中の失敗 | `stopTransfer()` でリセットし、再度開始(処理済み分は自動でスキップされる) |
| トリガーが作れない旨のエラー | トリガー 20 個上限 | エディタ「トリガー」画面で不要なトリガーを削除 |
| 何も起きずに終わる(`再開する処理はありません`) | 完了済み、または `stopTransfer` 済み | 新しく開始するなら `startTransfer()` |

## 5.9 FAQ

**Q. 間違って譲渡してしまった。戻せる?**
A. 新しいオーナー(譲渡先)のアカウントで逆方向に譲渡し直すしかありません。このリスクを避けるための DRY RUN です。譲渡先が協力してくれる状況なら、本ツールを譲渡先のアカウントにデプロイして逆向きに実行する、という手もあります。

**Q. 特定のフォルダだけ除外したい**
A. 現在の実装に除外機能はありません。運用でカバーするなら「除外したいものを対象フォルダの外へ移動してから実行」が簡単です。実装で対応するなら `runTreeBatch()` のサブフォルダ追加箇所に除外リスト判定を足すのが素直です(発展課題として[付録 E](./06-appendix.md#e-発展課題)参照)。

**Q. 共有ドライブのファイルはどうなる?**
A. 共有ドライブのアイテムには個人のオーナーがいないため、所有者チェックで自動的にスキップされます(何も起きません)。

**Q. ショートカットはどうなる?**
A. ショートカット自体も「自分がオーナーの 1 ファイル」なので、所有権が譲渡されます。リンク先の実体は別ファイルであり、実体のオーナーが自分なら実体側も(走査範囲にあれば)個別に処理されます。

**Q. 実行中に新しいファイルを作ったら?**
A. 走査済みの場所に後から作られたファイルは今回の実行では拾われません。完了後にもう一度実行すれば拾えます。引き継ぎ作業中は新規作成を控えるのが無難です。
