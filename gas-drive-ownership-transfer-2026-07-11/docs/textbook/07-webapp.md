# 第 7 章: Web アプリ UI(ブラウザから実行する)

Apps Script エディタを開かなくても、**ブラウザのボタン操作**で「規模確認 → 開始 → 進捗確認 → 停止」ができる Web アプリ UI を用意しています。実体は `src/webapp.ts`(サーバー側)と `src/index.html`(画面)です。

> この章の最重要テーマは **「誰がこのページを開けて、誰の権限で動くのか」**(= 共有範囲)です。ここを誤ると「他人があなたのファイルの譲渡を開始できるボタン」を配ってしまうことになるため、7.3 節は必ず読んでください。

## 7.1 何ができるか

ページには次の要素が並びます。

| UI 要素 | 対応する処理 |
| --- | --- |
| 設定カード | 実行者・譲渡先・対象フォルダ・モード(DRY RUN / 本番)の表示。**本番モードのときは赤い警告**が出る |
| 「所有アイテム数を確認」 | `countOwnedFiles()` 相当の概算を表示 |
| 「ツリー走査で開始」「検索走査で開始」 | 確認ダイアログ(本番時は二重確認)を経て開始 |
| 「進捗を更新」 | 保存されている進捗サマリを再取得(実行中は 20 秒ごとに自動更新) |
| 「停止(リセット)」 | `stopTransfer()` 相当 |

開始ボタンを押すと、**最初のバッチだけ 45 秒**で切り上げてすぐ画面に結果を返し、残りは通常どおりサーバー側のトリガー連鎖(1 バッチ 4.5 分)で自動継続します。ページを閉じても処理は止まりません。

<details>
<summary>📘 用語解説: HtmlService</summary>

GAS で HTML ページを配信するための組み込みサービスです。`HtmlService.createHtmlOutputFromFile('index')` のようにプロジェクト内の HTML ファイルを返すと、Google がホスティングする URL でページとして表示されます。ページは Google の安全なサンドボックス(別ドメインの iframe)内で描画されます。

</details>

<details>
<summary>📘 用語解説: doGet / doPost</summary>

Web アプリとしてデプロイされた GAS プロジェクトで、URL にアクセス(GET)があったときに自動で呼ばれる特別な関数が `doGet` です(フォーム送信などの POST は `doPost`)。この関数が返す HTML が、ユーザーのブラウザに表示されます。

</details>

<details>
<summary>📘 用語解説: google.script.run</summary>

HtmlService のページ内 JavaScript から、**サーバー側(GAS)の関数を呼び出す**ための仕組みです。`google.script.run.withSuccessHandler(fn).webAppGetStatus()` のように書くと、GAS 側の `webAppGetStatus()` が実行され、戻り値がブラウザ側の `fn` に渡ります。呼び出しは非同期で、認証は Google が自動処理します。

</details>

## 7.2 デプロイ手順

1. コードを push しておく(`mise run push`)
2. Apps Script エディタ右上の **「デプロイ」→「新しいデプロイ」**
3. 歯車アイコンから種類 **「ウェブアプリ」** を選択
4. 説明(任意)、そして次の 2 つの設定を選んで「デプロイ」
   - **次のユーザーとして実行**: `自分`(既定のまま)
   - **アクセスできるユーザー**: `自分のみ`(既定のまま)
5. 表示された **ウェブアプリの URL**(`https://script.google.com/.../exec`)を開く

この 2 つの既定値は、マニフェスト(`src/appsscript.json`)の `webapp` 設定として宣言してあります。

```json
"webapp": {
  "executeAs": "USER_DEPLOYING",
  "access": "MYSELF"
}
```

<details>
<summary>📘 用語解説: デプロイとバージョン</summary>

GAS の「デプロイ」は、その時点のコードの**スナップショット(バージョン)を固定して公開**する操作です。デプロイ後にコードを push しても、`/exec` の URL で動くのは固定されたバージョンのままです。更新を反映するには「デプロイを管理」→ 鉛筆アイコン →「新バージョン」で更新します。なお編集者専用の `/dev` URL は常に最新コードで動くため、開発中の動作確認にはこちらが便利です。

</details>

## 7.3 ⚠️ 共有範囲(アクセス設定)を完全に理解する

Web アプリの共有範囲は、Drive のファイル共有とは**まったく別の仕組み**で、デプロイ時の 2 軸の組み合わせだけで決まります。

### 軸 1: 「次のユーザーとして実行」(executeAs)

| 設定値 | 意味 | このツールでの意味 |
| --- | --- | --- |
| 自分(`USER_DEPLOYING`) | 誰がページを開いても、**デプロイした人の権限**で実行 | 譲渡されるのは常に**デプロイした人のファイル** |
| アクセスしているユーザー(`USER_ACCESSING`) | ページを開いた**本人の権限**で実行(初回に本人の承認画面が出る) | 譲渡されるのは**開いた人自身のファイル** |

### 軸 2: 「アクセスできるユーザー」(access)

| 設定値 | ページを開ける人 |
| --- | --- |
| 自分のみ(`MYSELF`) | デプロイした本人だけ。他人は URL を知っていても拒否される |
| ドメイン内の全員(`DOMAIN`) | 同じ Google Workspace 組織(例: `@3-shake.com`)のログインユーザー |
| Google アカウントを持つ全員(`ANYONE`) | Google にログインしていれば誰でも |
| 全員(`ANYONE_ANONYMOUS`) | ログイン不要で誰でも(このツールでは使わない) |

アクセス判定は **Google のログインで強制**されます。URL(`https://script.google.com/a/macros/<ドメイン>/s/<デプロイID>/exec`)が漏れても、設定範囲外の人には拒否ページが表示されるだけです。逆に言うと、**URL を秘密にすることは共有範囲の管理ではありません**。設定で守ります。

また、次の 2 点を混同しないでください。

- **スクリプトプロジェクト自体の Drive 共有**(エディタを開いて編集できる人)と、**Web アプリのアクセス設定**(ページを使える人)は独立しています。プロジェクトを共有していなくても、アクセス設定次第でページは使えます
- プロジェクトの**編集者**は `/dev` URL(最新コードで動く開発用)にもアクセスできます。編集権限の共有 = 実行の共有でもある点に注意

### 安全な組み合わせと危険な組み合わせ

| executeAs × access | 評価 | 説明 |
| --- | --- | --- |
| 自分 × 自分のみ | ✅ **既定・推奨** | 自分専用の操作パネル。リスク最小 |
| アクセスユーザー × ドメイン内 | ✅ 用途次第で推奨 | 「開いた人が自分自身のファイルを譲渡する」社内セルフサービスツールになる(7.4 節) |
| 自分 × ドメイン内/全員 | ❌ **絶対に避ける** | 組織の誰でも「**あなたの**ファイルの一括譲渡」を開始できるボタンになる |
| アクセスユーザー × 全員(匿名) | ❌ 不成立 | 匿名では本人の権限が存在しないため、この組み合わせ自体が機能しない |

> ⚠️ 本プロジェクトの譲渡ロジックは「実行している権限のユーザー(`Session.getEffectiveUser()`)が所有するもの」だけを対象にするため、危険な組み合わせでも**他人のファイルが譲渡されることはありません**。危険なのは常に「実行者(= 自分)のファイルが、他人の操作で譲渡されうる」ことです。

## 7.4 社内セルフサービス型にしたい場合

「退職者が各自でこのページを開き、自分のファイルを後任へ譲渡する」運用にする場合は、次のように変更してデプロイします。

1. `src/appsscript.json` を変更して push

```json
"webapp": {
  "executeAs": "USER_ACCESSING",
  "access": "DOMAIN"
}
```

2. 新しいデプロイを作成(設定欄も「アクセスしているユーザー」「(組織名)内の全員」になっていることを確認)
3. 利用者に URL を配布

利用者側の見え方:

- 初回アクセス時に、**その人自身の承認画面**(Drive の管理・トリガー作成などのスコープ)が出ます。組織内アプリなので「未確認アプリ」警告は通常出ません(組織のポリシーによります)
- 画面の「実行者」にはその人自身のメールアドレスが表示され、譲渡されるのはその人が所有するファイルだけです
- 進捗の保存先(スクリプトプロパティ)は**スクリプトごとに 1 つ**の共有領域なので、**複数人が同時に実行すると先行ジョブが優先され、後から開始しようとした人は「未完了の処理が残っています」エラーになります**。大人数で使う場合は「1 人ずつ順番に」の運用ルールにするか、保存先をユーザープロパティ(`getUserProperties()`)へ変える改造が必要です(発展課題)

<details>
<summary>📘 用語解説: スクリプトプロパティとユーザープロパティ</summary>

`PropertiesService` には 3 種類の保存領域があります。**スクリプトプロパティ**は全利用者で共有される領域、**ユーザープロパティ**は「スクリプト × 利用者」ごとに分かれる領域、**ドキュメントプロパティ**はバインドされたドキュメントごとの領域です。本ツールは 1 人で使う前提でスクリプトプロパティを使っていますが、セルフサービス化して同時実行を許すならユーザープロパティへの変更が適しています(あわせて LockService やトリガーの分離も検討が必要です)。

</details>

## 7.5 コードの読みどころ

### webapp.ts(サーバー側)

- `doGet()`: `index.html` を返すだけの入口です
- `webAppGetStatus()`: 設定と保存済み進捗を、HTML 側に渡せる素朴なオブジェクト(`WebAppStatus` 型)に詰め替えます。`google.script.run` で渡せるのは JSON にできる値だけなので、`TransferState` をそのまま返さず必要な数値だけを抜き出しています
- `webAppStartTree()` / `webAppStartSearch()`: 中身は CLI 版と同じ `startTransferWithStrategy()` ですが、第 2 引数で**最初のバッチの時間予算を 45 秒に短縮**しています

```typescript
const WEBAPP_FIRST_BATCH_MS = 45 * 1000;

function webAppStartTree(): WebAppStatus {
  startTransferWithStrategy('tree', WEBAPP_FIRST_BATCH_MS);
  return webAppGetStatus();
}
```

なぜ短縮するのか: `google.script.run` の呼び出しは、サーバー側の関数が終わるまで完了しません。エディタ実行と同じ 4.5 分バッチにすると、**ボタンを押してから 4.5 分間ブラウザが応答待ち**になってしまいます。そこで最初だけ 45 秒で切り上げて素早く画面に進捗を返し、続きはいつも通りトリガーに任せる、という設計です(2 バッチ目以降は 4.5 分に戻ります)。

### index.html(ブラウザ側)

- ボタン → `google.script.run.withSuccessHandler(...).webAppXxx()` という呼び出しの束です
- 開始時は `confirm()` ダイアログ、さらに**本番モードなら二重確認**を挟みます
- 実行中は 20 秒ごとに `webAppGetStatus()` をポーリングして進捗表を自動更新します
- エラー(ロック取得失敗・未完了状態あり・設定不備など)は `withFailureHandler` で受けて赤字で表示します

## 7.6 運用上の注意まとめ

- コードを更新したら **push だけでなく「デプロイを管理 → 新バージョン」**まで行わないと `/exec` に反映されない(7.2 節のアコーディオン参照)
- Web UI からでも安全フローは同じ: **DRY RUN で確認 → config.ts を本番に変えて push(+新バージョン)→ 本番実行**
- 細かいログ(1 件ごとの譲渡結果)は Web UI には出しません。Apps Script エディタの「実行数」から確認してください(第 5 章 5.5)
