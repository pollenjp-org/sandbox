# 第 7 章: Web アプリ UI(セルフサービス型)

Apps Script エディタを開かなくても、**ブラウザのフォームとボタン**で「規模確認 → 開始 → 進捗確認 → 停止」ができる Web アプリ UI を用意しています。実体は `src/webapp.ts`(サーバー側)と `src/index.html`(画面)です。

本プロジェクトの Web アプリは **セルフサービス型** の設計です。ページを開いた**本人の権限**でスクリプトが動き、本人が入力した譲渡先へ、**本人が所有するファイルだけ**を譲渡します。進捗・停止・再開トリガーもすべて本人専用で、複数人が同時に使っても互いに干渉しません(第 3 章のユーザープロパティ / ユーザーロック設計)。

> この章の最重要テーマは **「誰がこのページを開けて、誰の権限で動くのか」**(= 共有範囲)です。7.3 節は必ず読んでください。なお、シンプルに社内配布したいだけなら[第 8 章のスプレッドシート UI](./08-spreadsheet.md)の方が扱いやすいことが多いです。

## 7.1 何ができるか

ページには次の要素が並びます。

| UI 要素 | 内容 |
| --- | --- |
| 実行設定フォーム | **譲渡先メールアドレス**(必須)/ **対象フォルダ ID**(空欄 = マイドライブ全体。URL 貼り付け可)/ **モード**(DRY RUN / 本番)を利用者ごとに入力 |
| 「所有アイテム数を確認」 | 自分の所有アイテム数の概算を表示 |
| 「ツリー走査で開始」「検索走査で開始」 | 確認ダイアログ(本番は二重確認)を経て開始 |
| 「進捗を更新」 | 自分の進捗サマリを再取得(実行中は 20 秒ごとに自動更新) |
| 「停止(リセット)」 | 自分の処理の中止 |

開始ボタンを押すと、**最初のバッチだけ短時間(`CONFIG.uiFirstBatchMs` = 45 秒)**で切り上げてすぐ画面に結果を返し、残りは通常どおりサーバー側のトリガー連鎖(1 バッチ 4.5 分)で自動継続します。ページを閉じても処理は止まりません。

<details>
<summary>📘 用語解説: HtmlService</summary>

GAS で HTML ページを配信するための組み込みサービスです。`HtmlService.createHtmlOutputFromFile('index')` のようにプロジェクト内の HTML ファイルを返すと、Google がホスティングする URL でページとして表示されます。ページは Google の安全なサンドボックス(別ドメインの iframe)内で描画されます。

</details>

<details>
<summary>📘 用語解説: doGet / doPost</summary>

Web アプリとしてデプロイされた GAS プロジェクトで、URL にアクセス(GET)があったときに自動で呼ばれる特別な関数が `doGet` です(フォーム送信などの POST は `doPost`)。この関数が返す HTML が、ユーザーのブラウザに表示されます。

</details>

<details>
<summary>📘 用語解説: google.script.run</summary>

HtmlService のページ内 JavaScript から、**サーバー側(GAS)の関数を呼び出す**ための仕組みです。`google.script.run.withSuccessHandler(fn).webAppGetStatus()` のように書くと、GAS 側の `webAppGetStatus()` が実行され、戻り値がブラウザ側の `fn` に渡ります。呼び出しは非同期で、認証は Google が自動処理します。

</details>

## 7.2 デプロイ手順

1. コードを push しておく(`mise run push`)
2. Apps Script エディタ右上の **「デプロイ」→「新しいデプロイ」**
3. 歯車アイコンから種類 **「ウェブアプリ」** を選択
4. 説明(任意)、そして次の 2 つの設定を確認して「デプロイ」
   - **次のユーザーとして実行**: `ウェブアプリケーションにアクセスしているユーザー`
   - **アクセスできるユーザー**: `Google アカウントを持つ全員`
5. 表示された **ウェブアプリの URL**(`https://script.google.com/.../exec`)を利用者に配布

この 2 つの既定値は、マニフェスト(`src/appsscript.json`)の `webapp` 設定として宣言してあります。

```json
"webapp": {
  "executeAs": "USER_ACCESSING",
  "access": "ANYONE"
}
```

<details>
<summary>📘 用語解説: デプロイとバージョン</summary>

GAS の「デプロイ」は、その時点のコードの**スナップショット(バージョン)を固定して公開**する操作です。デプロイ後にコードを push しても、`/exec` の URL で動くのは固定されたバージョンのままです。更新を反映するには「デプロイを管理」→ 鉛筆アイコン →「新バージョン」で更新します。なお編集者専用の `/dev` URL は常に最新コードで動くため、開発中の動作確認にはこちらが便利です。

</details>

## 7.3 ⚠️ 共有範囲(アクセス設定)を完全に理解する

Web アプリの共有範囲は、Drive のファイル共有とは**まったく別の仕組み**で、デプロイ時の 2 軸の組み合わせだけで決まります。

### 軸 1: 「次のユーザーとして実行」(executeAs)

| 設定値 | 意味 | このツールでの意味 |
| --- | --- | --- |
| 自分(`USER_DEPLOYING`) | 誰がページを開いても、**デプロイした人の権限**で実行 | 譲渡されるのは常に**デプロイした人のファイル** |
| **アクセスしているユーザー(`USER_ACCESSING`)【採用】** | ページを開いた**本人の権限**で実行(初回に本人の承認画面が出る) | 譲渡されるのは**開いた人自身のファイル** |

### 軸 2: 「アクセスできるユーザー」(access)

| 設定値 | ページを開ける人 |
| --- | --- |
| 自分のみ(`MYSELF`) | デプロイした本人だけ |
| ドメイン内の全員(`DOMAIN`) | 同じ Google Workspace 組織のログインユーザー |
| **Google アカウントを持つ全員(`ANYONE`)【採用】** | Google にログインしていれば誰でも |
| 全員(`ANYONE_ANONYMOUS`) | ログイン不要で誰でも(このツールでは使わない) |

アクセス判定は **Google のログインで強制**されます。URL(`https://script.google.com/a/macros/<ドメイン>/s/<デプロイID>/exec`)が漏れても、設定範囲外の人には拒否ページが表示されるだけです。逆に言うと、**URL を秘密にすることは共有範囲の管理ではありません**。設定で守ります。

また、次の 2 点を混同しないでください。

- **スクリプトプロジェクト自体の Drive 共有**(エディタを開いて編集できる人)と、**Web アプリのアクセス設定**(ページを使える人)は独立しています
- プロジェクトの**編集者**は `/dev` URL(最新コードで動く開発用)にもアクセスできます。編集権限の共有 = 実行の共有でもある点に注意

### 組み合わせの評価

| executeAs × access | 評価 | 説明 |
| --- | --- | --- |
| アクセスユーザー × Google アカウントを持つ全員 | ✅ **本プロジェクトの採用構成** | 開いた本人のファイルを、本人の承認のもとで処理。他人のファイルには構造的に触れない |
| アクセスユーザー × ドメイン内 | ✅ より堅い選択肢 | 利用者を組織内に限定したい場合はこちらに変更(`access: "DOMAIN"`) |
| 自分 × 自分のみ | ✅ 個人用 | 自分専用の操作パネルにしたい場合(`USER_DEPLOYING` + `MYSELF`) |
| 自分 × ドメイン内/全員 | ❌ **絶対に避ける** | 誰でも「**デプロイした人の**ファイルの一括譲渡」を開始できるボタンになる |
| アクセスユーザー × 全員(匿名) | ❌ 不成立 | 匿名では本人の権限が存在しないため機能しない |

### 「全員」にしても、社外の人は実質使えない

採用構成は「Google アカウントを持つ全員」がページを開けますが、**組織外のユーザーが実際に使える可能性は低い**ことを知っておいてください。

1. このスクリプトはフル Drive スコープ(`auth/drive`)を要求します。これは Google の分類で「**制限付き(restricted)スコープ**」で、**組織内の利用者には内部アプリとして普通に承認画面が出ます**が、**組織外の Google アカウントには「未確認アプリ」のブロック/警告**が立ちはだかります。外部公開でまともに使える状態にするには Google の OAuth 検証(セキュリティ審査込み)が必要です
2. 仮に承認できても、Workspace の所有権譲渡は**同一ドメイン間しか成立しない**ため、社外ユーザーの譲渡は `setOwner()` でエラーになります

つまり実質的な利用者は同一ドメインの人に絞られます。それを設定として明示したい場合は `access: "DOMAIN"` に変えてください(挙動は同じで、意図が読み取りやすくなります)。

## 7.4 複数人で同時に使える理由(セルフサービスの土台)

Web アプリを複数人に配ると「他人と混線しないか」が問題になります。本プロジェクトは以下の分離で、利用者ごとの実行を完全に独立させています(第 3 章参照)。

| リソース | 使っているもの | 分離の単位 |
| --- | --- | --- |
| 進捗チェックポイント | ユーザープロパティ(`getUserProperties()`) | スクリプト × 利用者 |
| 排他ロック | ユーザーロック(`getUserLock()`) | 利用者ごと |
| 再開トリガー | 作成した利用者のトリガーとして登録される | 利用者ごと(上限 20 個も利用者単位) |
| 実行権限 | `USER_ACCESSING`(本人として実行) | 利用者ごと |

この構成により、**他人の進捗を見る・止める・上書きすることはできません**。「停止(リセット)」ボタンが消せるのも自分の実行だけです。

<details>
<summary>📘 用語解説: スクリプトプロパティとユーザープロパティ</summary>

`PropertiesService` には 3 種類の保存領域があります。**スクリプトプロパティ**は全利用者で共有される領域、**ユーザープロパティ**は「スクリプト × 利用者」ごとに分かれる領域、**ドキュメントプロパティ**はバインドされたドキュメントごとの領域です。本ツールは複数人利用を前提に**ユーザープロパティを採用**しています。仮にスクリプトプロパティを使うと、全員で 1 つの進捗枠を取り合い、他人のジョブをリセットできてしまいます。

</details>

## 7.5 コードの読みどころ

### webapp.ts(サーバー側)

- `doGet()`: `index.html` を返すだけの入口です
- `webAppGetStatus()`: 本人の保存済み進捗を、HTML 側に渡せる素朴なオブジェクト(`WebAppStatus` 型)に詰め替えます。`google.script.run` で渡せるのは JSON にできる値だけなので、`TransferState` をそのまま返さず必要な値だけを抜き出しています
- `webAppStartTree(newOwnerEmail, rootFolderId, dryRun)` / `webAppStartSearch(...)`: 画面の入力値を `TransferStartOptions` に正規化して、CLI 版と同じ `startTransferWithStrategy()` に渡します

```typescript
function normalizeWebAppOptions(newOwnerEmail, rootFolderId, dryRun) {
  return {
    maxRuntimeMs: CONFIG.uiFirstBatchMs,          // 最初のバッチだけ 45 秒で切り上げ
    newOwnerEmail: String(...).trim(),
    rootFolderId: normalizeFolderIdInput(...),    // フォルダ URL 貼り付けにも対応
    dryRun: dryRun === false ? false : true,      // 想定外の値は安全側(DRY RUN)に倒す
  };
}
```

なぜ最初のバッチを短縮するのか: `google.script.run` の呼び出しは、サーバー側の関数が終わるまで完了しません。エディタ実行と同じ 4.5 分バッチにすると、**ボタンを押してから 4.5 分間ブラウザが応答待ち**になってしまいます。そこで最初だけ 45 秒で切り上げて素早く画面に進捗を返し、続きはいつも通りトリガーに任せる、という設計です(2 バッチ目以降は 4.5 分に戻ります)。

`dryRun === false ? false : true` という一見回りくどい書き方は、**「明示的に false と言われたときだけ本番」**という安全側の倒し方です。クライアント側のバグで `undefined` が飛んできても本番実行にはなりません。

### index.html(ブラウザ側)

- フォームの入力値を検証(譲渡先の形式チェック)してから、`google.script.run.withSuccessHandler(...).webAppStartTree(入力値...)` を呼ぶ構成です
- 開始時は `confirm()` ダイアログに**入力した譲渡先を表示**し、さらに**本番モードなら二重確認**を挟みます
- 実行中は 20 秒ごとに `webAppGetStatus()` をポーリングして進捗表を自動更新します
- エラー(ロック取得失敗・未完了状態あり・設定不備など)は `withFailureHandler` で受けて赤字で表示します

## 7.6 運用上の注意まとめ

- コードを更新したら **push だけでなく「デプロイを管理 → 新バージョン」**まで行わないと `/exec` に反映されない(7.2 節のアコーディオン参照)
- 安全フローは Web でも同じ: **DRY RUN で確認 → モードを「本番」に切り替えて実行**(モードは画面で選ぶので push は不要)
- 細かいログ(1 件ごとの譲渡結果)は Web UI には出しません。各利用者が自分の Apps Script の「実行数」から確認できます。**台帳として記録を残したい運用なら、第 8 章のスプレッドシート UI を使ってください**
